# テスト仕様書スキル

`docs/manual_test.md` を対象に、テストケースの新規作成・追記・修正・レビューを行う。

## 呼び出し方

- `/test-spec` — 追記・修正の指示を会話で受け取る
- `/test-spec review` — `manual_test.md` 全体をレビューして問題箇所を報告する
- `/test-spec <セクション名>` — 指定セクションのテストケースを新規作成する

---

## フォーマット

`manual_test.md` のセクション構成と表形式を必ず踏襲する。

```markdown
## N. セクション名（`/Controller`）

| # | 操作 | 確認内容 | 結果 |
|---|------|----------|------|
| N-1 | ユーザーが行う操作を書く | 期待される結果を書く | [ ] |
```

- `操作` 列: ユーザーが画面上で行う手順を具体的に書く（クリックするボタン名・入力値を明記）
- `確認内容` 列: ユーザーが目で見て確認できる結果を書く。**CSS セレクターや HTML 要素名は書かない**
- `結果` 列: 常に `[ ]`（テスト実施者が手動でチェックする）

---

## 品質ルール

### 1. 条件付き挙動は分割する

**NG（やってはいけない書き方）**:
> 確認ダイアログが表示される（〜の場合は出ない場合もある）

**OK（正しい書き方）**:
テストが実行される時点のシステム状態に合わせて、別々のテストケースとして書く。

```
| 3-3 | 掛け率を変更 → 保存 | この時点では取引先が未登録のため、確認ダイアログなしで保存される |
...（セクション4: 取引先を登録してから）
| 4-4 | 掛け率を変更 → 保存 | ABC商店がこの商品を取り扱っているため、「標準値のみ更新」「取引先設定も一括更新」の選択ダイアログが表示される |
```

「〜場合もある」「〜の可能性がある」は書かない。その時点で何が起きるかを断言する。

### 2. 前提状態を明示する

確認内容に「この時点では〜のため」と前提を添えると、なぜその結果になるかが明確になる。
特に、同じ操作でも前後の状態によって結果が変わる場合は必ず書く。

### 3. 確認内容は人間の言葉で書く

**NG**: `td[rowspan]` 要素が存在する  
**OK**: 同じ商品名の複数行がまとめて表示される

**NG**: `.readonly-banner` が表示され保存ボタンがない  
**OK**: 「翌月分が登録済みのため編集できません」バナーが表示される。保存ボタンがない

### 4. 具体的な数値を書く

**NG**: 正しい売上額が表示される  
**OK**: 売上点数 = 0 + 10 − 8 = **2**、売上額 = 2 × 1500 = **3,000**

計算結果や期待値が確定できる場合は、実際の数値まで書く。

### 5. 後処理テストを忘れない

テストデータを変更するケース（編集・削除）では、後続テストへの影響を考慮した「元に戻す」テストケースをセットで用意する。

### 6. テストの順序依存に注意する

テストは上から順に実行し、前のテストで登録したデータが残った状態で次を実行する。
新しいケースを追加するときは「この時点で何が登録済みか」を必ず確認してから書く。

## レビュー時のチェックリスト

`/test-spec review` を実行したときは以下を確認して報告する。

- [ ] 「〜場合もある」「〜の可能性がある」などの曖昧な表現がないか
- [ ] 確認内容に CSS セレクター・HTML タグ・クラス名が混入していないか
- [ ] 確認内容が「表示される」だけで終わっていて数値が省略されていないか
- [ ] 変更系テストに対応する「元に戻す」ケースがあるか
- [ ] 前提状態が変わるのに確認内容が変わっていないケースがないか（状態依存の挙動の見落とし）
